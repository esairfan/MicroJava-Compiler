\begin{tcolorbox}[infobox, title={Abstract}]
This report details the design and implementation of a multi-stage syntax validator and execution engine for the \textbf{MicroJava} language. The compiler incorporates five core phases: double-buffered lexical analysis, recursive descent parsing with dynamic ASCII tree visualization, chained-scope symbol table management, LL(1) non-recursive predictive parsing using an explicit stack, and Canonical LR(1) bottom-up parsing. We explore grammar transformations required to resolve left recursion and left factoring for LL(1) predictive parsing. We also justify the choice of Canonical LR(1) for bottom-up parsing and present end-to-end execution traces mapping input source code through every compilation layer.
\end{tcolorbox}

\section{Introduction and Language Overview}

\subsection{Introduction to MicroJava}
MicroJava is a subset of Java designed for compiler construction courses. It is a strongly-typed, object-oriented language that supports basic data types (integers and characters), constants, arrays, static class-like structures, methods, nested scopes, and control flow statements such as conditional execution and loops.

\subsection{Key Implementation Features}
The compiler implementation incorporates the following architectural highlights:

\begin{itemize}
  \item \textbf{C++17 Compiler Core:} Highly performant binary execution layer supporting five synchronized analysis phases.
  \item \textbf{Single-Error Termination:} All three parser engines stop execution immediately at the first lexical, syntax, or semantic error, preventing cascading failures.
  \item \textbf{Visual Representation:} The Recursive Descent parser outputs a colored ASCII concrete syntax tree illustrating exact scope nesting and grammar expansions.
\end{itemize}

\section{MicroJava EBNF Grammar Specification}

The parser expects code written in accordance with the Context-Free Grammar (CFG) rules below:

\begin{lstlisting}[language=HTML, caption={MicroJava EBNF Grammar}]
Program    = "program" ident {ConstDecl | VarDecl | ClassDecl} "{" {MethodDecl} "}".
ConstDecl  = "final" Type ident "=" (number | charConst) ";".
VarDecl    = Type ident {"," ident} ";".
ClassDecl  = "class" ident "{" {VarDecl} "}".
MethodDecl = (Type | "void") ident "(" [FormPars] ")" {VarDecl} Block.
FormPars   = Type ident {"," Type ident}.
Type       = ident ["[" "]"].
Block      = "{" {Statement} "}".
Statement  = Designator ("=" Expr | ActPars) ";"
           | "if" "(" Condition ")" Statement ["else" Statement]
           | "while" "(" Condition ")" Statement
           | "return" [Expr] ";"
           | "read" "(" Designator ")" ";"
           | "print" "(" Expr ["," number] ")" ";"
           | Block
           | ";".
ActPars    = "(" [Expr {"," Expr}] ")".
Condition  = Expr Relop Expr.
Relop      = "==" | "!=" | ">" | ">=" | "<" | "<=".
Expr       = ["-"] Term {Addop Term}.
Addop      = "+" | "-".
Term       = Factor {Mulop Factor}.
Mulop      = "*" | "/" | "%".
Factor     = Designator [ActPars]
           | number
           | charConst
           | "new" ident ["[" Expr "]"]
           | "(" Expr ")".
Designator = ident {"." ident | "[" Expr "]"}.
\end{lstlisting}

% ─── LEFT RECURSION & LEFT FACTORING ──────────────────────────────────────────
\subsection{Left Recursion and Left Factoring Solutions}

To build a top-down parser such as LL(1) or Recursive Descent, the grammar must be free from:

\begin{enumerate}
  \item \textbf{Left Recursion:} Rules of the form $A \rightarrow A\,\alpha \mid \beta$ cause infinite recursion in top-down parsers.
  \item \textbf{Common Prefixes:} Rules of the form $A \rightarrow \alpha\,\beta_1 \mid \alpha\,\beta_2$ prevent deterministic single-token lookahead.
\end{enumerate}

\subsubsection{Eliminating Left Recursion}

Direct left recursion is eliminated using the standard transformation:

\begin{tcolorbox}[colback=LightBlack!30, colframe=PrimaryBlack, arc=3pt, boxrule=0.6pt]
\textbf{General Rule:}
\[
  A \;\rightarrow\; A\,\alpha \;\mid\; \beta
  \qquad\Longrightarrow\qquad
  A \;\rightarrow\; \beta\,A' \qquad A' \;\rightarrow\; \alpha\,A' \;\mid\; \varepsilon
\]
\end{tcolorbox}

\medskip
\textbf{Example — Expression Grammar:}

\noindent\textit{Original (left-recursive):}
\[
  \textit{Expr} \;\rightarrow\; \textit{Expr}\;\textit{Addop}\;\textit{Term} \;\mid\; \textit{Term}
\]

\noindent\textit{Transformed:}
\[
  \textit{Expr}  \;\rightarrow\; \textit{Term}\;\textit{Expr}'
  \qquad
  \textit{Expr}' \;\rightarrow\; \textit{Addop}\;\textit{Term}\;\textit{Expr}' \;\mid\; \varepsilon
\]

\subsubsection{Left Factoring}

Common prefixes are factored out to allow deterministic parsing:

\begin{tcolorbox}[colback=LightBlack!30, colframe=PrimaryBlack, arc=3pt, boxrule=0.6pt]
\textbf{General Rule:}
\[
  A \;\rightarrow\; \alpha\,\beta_1 \;\mid\; \alpha\,\beta_2
  \qquad\Longrightarrow\qquad
  A \;\rightarrow\; \alpha\,A' \qquad A' \;\rightarrow\; \beta_1 \;\mid\; \beta_2
\]
\end{tcolorbox}

\medskip
\textbf{Example — Designator Tail:}

\noindent\textit{Original:}
\[
  \textit{Designator} \;\rightarrow\; \textit{ident}\,.\,\textit{ident}
    \;\mid\; \textit{ident}\,[\,\textit{Expr}\,]
    \;\mid\; \textit{ident}
\]

\noindent\textit{Transformed:}
\[
  \textit{Designator}      \;\rightarrow\; \textit{ident}\;\textit{Designator\_Tail}
\]
\[
  \textit{Designator\_Tail} \;\rightarrow\;
    \mathtt{.}\;\textit{ident}\;\textit{Designator\_Tail}
    \;\mid\; \mathtt{[}\;\textit{Expr}\;\mathtt{]}\;\textit{Designator\_Tail}
    \;\mid\; \varepsilon
\]

\section{Theoretical Logic and Implementation Details}

\subsection{Lexical Analyzer (Lexer)}

\subsubsection{Theoretical Logic}
The lexer converts a sequence of characters into a sequence of tokens, reading from a file stream and partitioning them based on regular definitions of the vocabulary:

\begin{itemize}
  \item $\textit{Keyword} = \mathbf{program} \mid \mathbf{class} \mid \mathbf{void} \mid \mathbf{final} \mid \mathbf{new} \mid \mathbf{if} \mid \mathbf{else} \mid \mathbf{while} \mid \mathbf{read} \mid \mathbf{print} \mid \mathbf{return}$
  \item $\textit{Identifier} = \text{letter}\,(\text{letter} \mid \text{digit})^*$
  \item $\textit{Number} = \text{digit}^+$
  \item $\textit{CharConst} = \texttt{'}\;\text{any\_printable\_char}\;\texttt{'}$
\end{itemize}

It uses a \textbf{double-buffering scheme} to process characters efficiently and tracks character coordinates (line and column numbers) to pinpoint lexical errors.

\subsection{Recursive Descent Parser \& Tree Logic}

\subsubsection{Theoretical Logic}
The Recursive Descent (RD) parser is a top-down validation system that matches the token sequence by recursively calling parsing functions corresponding to each non-terminal.

\subsubsection{Tree Generation Logic}
The parser outputs a text-based Concrete Syntax Tree (CST) by tracking the call stack depth via a nesting counter:

\begin{itemize}
  \item When a parsing function (e.g., \texttt{parseVarDecl()}) is entered, \texttt{indentLevel} is incremented.
  \item A branch node name is printed prefixed by vertical lines ($\texttt{indentLevel} \times \texttt{|\ }$).
  \item When the function returns, \texttt{indentLevel} is decremented.
\end{itemize}

Because this tree corresponds exactly to the parser's active execution stack, it represents the leftmost derivation hierarchy of the input program.

\subsection{Non-Recursive LL(1) Stack Parser}

\subsubsection{Theoretical Logic}
The LL(1) parser uses a non-recursive predictive parsing model with an explicit stack. It looks up transitions in the 2-D parser table $M[A,\,a]$:

\begin{itemize}
  \item If the top of the stack is a terminal matching the lookahead token, it pops and advances.
  \item If the top is a non-terminal $A$, it queries $M[A, a]$ to retrieve rule $A \rightarrow Y_1 Y_2 \dots Y_k$, pops $A$, and pushes $Y_k, Y_{k-1}, \dots, Y_1$ onto the stack in reverse order.
\end{itemize}

\subsection{Canonical LR(1) Parser}

\subsubsection{Theoretical Logic and Choice Justification}
The LR(1) parser is a bottom-up shift-reduce syntax validator. \textbf{Canonical LR(1)} was selected over LR(0), SLR(1), or LALR(1) for the following reasons:

\begin{enumerate}
  \item \textbf{Explicit Lookahead Handling:} LR(1) embeds lookahead items $[A \rightarrow \alpha \cdot \beta,\; a]$ directly within states, resolving shift/reduce conflicts that SLR(1) cannot handle via FOLLOW sets alone.
  \item \textbf{No Conflict Merging:} Unlike LALR(1), which merges states with the same core items but different lookaheads, Canonical LR(1) keeps these states split, guaranteeing strict parsing correctness.
  \item \textbf{Handles Complex Subsets:} MicroJava contains nested expression boundaries and variable scopes where shift-reduce decisions depend heavily on the lookahead; LR(1) guarantees unambiguous transitions.
\end{enumerate}

\subsubsection{Standard Shift-Reduce Algorithm}
The parser maintains a stack of states $s_0\,s_1\,\dots\,s_m$ and a corresponding symbol stack $X_1\,X_2\,\dots\,X_m$. For each step, let $s$ be the top state and $a$ the current lookahead. It queries $\textsc{Action}[s,\,a]$:

\begin{enumerate}
  \item \textbf{Shift to state $s'$:} Push $a$ onto the symbol stack, push $s'$ onto the state stack, fetch the next token.
  \item \textbf{Reduce by $A \rightarrow \beta$:} Pop $r = |\beta|$ symbols and states. Let the new top state be $s''$. Query $s_{\text{new}} = \textsc{Goto}[s'',\,A]$; push $A$ and $s_{\text{new}}$.
  \item \textbf{Accept:} Terminate with success.
  \item \textbf{Error:} Halt immediately; print line and column number.
\end{enumerate}

\section{Compiler Error Handling and Halt-on-Error Semantics}

The compiler implements strict validation rules where \textbf{compilation halts immediately at the first syntax or semantic error}. When the parser detects an invalid statement or mismatched token:

\begin{itemize}
  \item It raises a \texttt{std::runtime\_error} inside the \texttt{reportError()} function.
  \item This prints the error details (with line/column coordinates) and aborts the parser call stack.
  \item The console displays all output generated \emph{prior} to the failure, giving a clear visual boundary between valid parsed structures and the error point.
\end{itemize}
